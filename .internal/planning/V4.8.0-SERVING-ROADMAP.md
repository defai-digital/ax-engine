# AX Engine v4.8.0 Serving Roadmap

Status: proposed for the v4.8.0 release cycle
Scope: serving-oriented AX Engine runtime improvements
License direction: Apache License, Version 2.0

## Context

AX Engine v4.8.0 will release more serving capability in the open-source
engine. The license moves from MIT to Apache-2.0 to better match a runtime
roadmap where serving features, memory behavior, scheduler behavior, and kernel
work may become long-lived public extension points.

The roadmap should preserve the existing project boundary:

- repo-owned MLX runtime claims must stay evidence-backed
- delegated `mlx_lm` and `llama.cpp` routes remain explicit compatibility
  routes, not AX-owned performance claims
- serving claims need online-serving artifacts, not only batch=1 decode tables
- high-risk kernel and quantization work should be promoted only after
  correctness, determinism, memory-pressure, and benchmark gates pass

## Direction 1: KV Cache Memory Layout

Decode is often memory-bandwidth-bound rather than compute-bound. If the KV
cache layout scatters reads, GPU time is lost to memory access rather than
model arithmetic.

Potential work:

- paged KV cache
- block-aligned KV layout
- per-layer KV locality optimization
- reduced KV copy and transpose
- cache reuse between speculative draft and target verification paths

Serving value:

- long-context stability
- multi-request serving throughput
- lower decode-time memory bandwidth pressure
- clearer prefix/cache reuse contracts

## Direction 2: Apple Unified Memory Advantage

Apple Silicon's unified memory model lets AX Engine avoid parts of the
traditional host-device copy model used by many GPU runtimes.

Potential work:

- zero-copy model weight mapping
- memory-mapped quantized weights
- direct Metal buffer reuse
- avoid unnecessary tensor materialization
- persistent buffers across requests

Serving value:

- faster cold start
- lower TTFT
- lower memory pressure
- better large-model loading behavior

## Direction 3: MoE Expert Locality Optimization

AX Engine already shows promising behavior on 35B-class MoE models, so expert
locality is a high-value serving direction.

Potential work:

- expert weight cache scheduling
- group tokens by expert
- reduce expert dispatch overhead
- prefetch likely expert weights
- fuse router and expert dispatch
- optimize top-k routing memory pattern

Serving value:

- lower routing overhead
- better sparse-model throughput
- stronger multi-request MoE behavior
- less memory traffic for cold or low-probability experts

## Direction 4: Speculative Decoding Software Tuning

Current benchmark evidence shows the largest user-visible decode gains come
from n-gram speculative decoding, not direct decode alone. AX Engine should
lean into the positioning of a fast speculative inference runtime.

Potential work:

- adaptive n-gram length
- dynamic draft window
- acceptance-rate predictor
- fallback threshold tuning
- prompt-pattern-aware speculation
- cache sharing between draft and verify paths

Serving value:

- better 8B/9B fallback behavior
- stronger coding and structured-output throughput
- fewer bad-speculation stalls
- clearer per-request speculation telemetry

## Direction 5: Kernel Fusion And Quantization Path

To build a durable advantage over reference runtimes, AX Engine needs selected
owned kernels where the benefit is large enough to justify the maintenance
cost.

Potential work:

- fused RMSNorm and matmul
- fused attention projection
- fused dequant and matmul
- group-wise quantization kernel optimization
- Apple AMX/Metal mixed path
- prepacked weight layout

Serving value:

- lower per-token launch overhead
- better quantized decode throughput
- lower memory bandwidth use
- a more defensible runtime moat once correctness gates are mature

## Promotion Gates

Each roadmap item should define the gate before public claims change:

- correctness parity against the reference model path
- deterministic replay or bounded stochastic variance
- benchmark artifacts for prefill, decode, TTFT, and online serving when
  serving behavior is claimed
- memory-pressure telemetry, including cold-start and long-context cases
- route identity in artifacts so delegated compatibility is not mixed with
  repo-owned runtime performance
- fallback behavior that is visible in telemetry and safe under low acceptance
  or unsupported cache layouts
