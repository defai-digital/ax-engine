# W1.3 C2 — cached_scalar in scale_hidden (2026-05-14)

PRD: `MLX-DECODE-SUBMIT-COST-PRD.md` W1.3
Audit: `MLX-DECODE-W13-CAST-RESHAPE-AUDIT.md` C2

## Change

`scale_hidden` was building a fresh `[1] f32` array and `astype`-ing it
to the target dtype on every call. Switched to the existing
`cached_scalar` helper (in mlx-sys/ops.rs) which dedups by
`(value_bits, dtype)`. The cold `scalar_like` path is retained for the
two non-hot-path callers (MoE router masking, test fixtures) where the
value is unique per call.

```rust
fn scale_hidden(hidden: &MlxArray, scale: f32) -> MlxArray {
    let s_arr = mlx_sys::ops::cached_scalar(scale, hidden.dtype());
    multiply(hidden, &s_arr, None)
}
```

`scale_hidden` is called 4× per Gemma 4 E2B decode step:

- forward_lazy_single: `hidden_states_scale` (39.19)
- compute_per_layer_inputs_arr: `sqrt(per_layer_dim)` (16.0)
- compute_per_layer_inputs_arr: `1/sqrt(hidden_size)` (~0.0255)
- compute_per_layer_inputs_arr: `FRAC_1_SQRT_2` (~0.7071)

All four values are constant per process, so the `cached_scalar` table
hits on every call after the first.

## A/B (M5 Max, gemma-4-e2b-it-4bit, 128/128, 3 reps)

| Metric | C3 baseline | C2 | Δ |
|---|---:|---:|---:|
| decode_tok_s (median) | 184.96 | 185.39 | +0.24% |
| direct_pipeline_wall_us / step | 5,343.51 | 5,334.63 | -8.88 µs |
| direct_pipeline_async_eval_wall_us / step | 4,917.10 | 4,908.55 | -8.55 µs |
| **direct_pipeline_op_count / step** | **1,769** | **1,765** | **-4** |

Op-count drop matches prediction exactly (4 astype dispatches eliminated,
one per scale site). The 8.55 µs/step async_eval reduction matches the
expected 4 × ~2 µs/op submit budget — the first per-step bucket-level
delta within audit-predicted noise.

Cross-family safety: `scale_hidden` is also used by Qwen / GLM paths.
The change is value-equivalent (cached_scalar internally calls
`from_raw_data` + `astype` exactly the same way as `scalar_like` did);
later calls just reuse the cached array. No numerical drift possible.
