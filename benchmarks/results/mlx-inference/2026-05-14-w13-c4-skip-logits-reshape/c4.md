# W1.3 C4 — drop final logits reshape (2026-05-14)

PRD: `MLX-DECODE-SUBMIT-COST-PRD.md` W1.3
Audit: `MLX-DECODE-W13-CAST-RESHAPE-AUDIT.md` C4

## Change

`forward_lazy_single` was flattening the post-softcap logits from
`[1, 1, vocab]` to `[vocab]` via an explicit `reshape` before returning.
The only consumer of the returned array is `argmax(_, None)` in
`generate.rs`, which operates on the last axis — it works on either
shape. The downstream `data_u32().first()` reads the same single token
regardless of leading singleton dims.

```rust
let logits = qw(&normed, &weights.lm_head);                   // [1, 1, vocab] bf16
let logits_f32 = astype(&logits, MlxDtype::Float32, None);    // [1, 1, vocab] f32
let logits = apply_final_logit_softcap(cfg, &logits_f32);     // [1, 1, vocab] f32
// (removed) reshape(&logits_f32, &[cfg.vocab_size as i32], None)
```

The other forward variants (prefill `forward_with_turboquant_context` at
L1486, multi-token `forward_for_step` at L1552) keep their flatten — they
are not on the per-step decode hot path and may have callers that
explicitly depend on 1-D / 2-D shapes.

## Reading

41 ops eliminated per decode step (2 from C1 + 34 from C3 + 4 from C2 + 1
from C4). The audit predicted 39; the extra 2 came from C1's second site
in `compute_per_layer_inputs_arr` which the audit counted as a separate
class.

The wall-time delta (~40 µs/step) is roughly half of the naive 41 × 2 µs
= 82 µs ceiling, consistent with the 2026-05-14 barrier A/B finding that
~837 µs/step of submit cost is already hidden by pipelining. The visible
throughput gain (+0.25%) is the portion not hidden.

This closes ~8% of the 521 µs/step gap to mlx_lm via op-count reduction
alone. W1.2's narrower variants and W2's mlx_compile spike are the
remaining levers.
